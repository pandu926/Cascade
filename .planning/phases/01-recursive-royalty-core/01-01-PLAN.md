  <behavior>
    - register: assigns next monotonic id starting at 1 (id 0 reserved/invalid); stores immutable Skill{creator, price, depIds[], depShares[], depth}; reverts if any depId == 0 or depId >= newId (enforces strictly-smaller / bottom-up, makes cycles impossible per CORE-02); reverts if Σ depShares > 10000 (CORE-01); computes depth = 1 + max(dep depths) and reverts if depth > 8 (CORE-04); emits SkillRegistered. depShares.length must equal depIds.length.
    - invoke: reverts unless msg.value == skill.price (exact); performs recursive top-down split — for the invoked skill, each declared dependency receives floor(amount * depShare / 10000) routed into that dependency's subtree (recursively applying the same split), and the current skill's creator receives the remainder of its own level; integer-division dust at every level rolls up so the invoked (top) skill's creator absorbs total dust (CORE: dust→top creator); ONLY updates balances[creator] (no external calls — CORE-05); emits RoyaltyAccrued per credited creator and Invoked once. Recursion bounded by the register-time depth cap (CORE-04).
    - claim: reads balances[msg.sender], zeroes it BEFORE transferring, sends via call, reverts on transfer failure; second claim sends 0 / no double-pay (CORE-06); emits Claimed.
    - Conservation invariant: Σ of all RoyaltyAccrued amounts for one invoke == price exactly.
  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller → Cascade.invoke | Untrusted payer sends native PHRS; amount and skillId are untrusted input |
| creator → Cascade.claim | Untrusted creator withdraws accrued balance |
| author → Cascade.register | Untrusted author declares price, deps, shares |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering / Elevation | claim() reentrancy | mitigate | Pull-payment; zero `balances[msg.sender]` BEFORE the external `call`; no fan-out during invoke (CORE-05). Test asserts no double-pay. |
| T-01-02 | Denial of Service | invoke fan-out blocked by a malicious payee | mitigate | No external calls during invoke — only internal balance accrual; a payee cannot block the tree. |
| T-01-03 | Denial of Service | unbounded recursion gas | mitigate | Depth cap = 8 enforced at register time; invoke recursion provably bounded (CORE-04); depth-cap revert test. |
| T-01-04 | Tampering | dependency cycle / forward reference | mitigate | Monotonic ids; dep must be strictly-smaller registered id; cycles impossible by construction (CORE-02). |
| T-01-05 | Information disclosure / Integrity | value loss via rounding | mitigate | Conservation invariant test (Σ accrued == price); dust deterministically credited to top creator. |
| T-01-06 | Tampering | over/underpayment | mitigate | Exact `msg.value == price` check; revert test. |
| T-01-07 | Repudiation | money flow not auditable | mitigate | Emit SkillRegistered/Invoked/RoyaltyAccrued/Claimed; Phase 4 reconstructs flow from logs. |
| T-01-SC | Tampering | dependency supply chain | accept | Only dep is forge-std (git, well-known foundry-rs); no npm/pip/cargo installs in this phase. |
</threat_model>
